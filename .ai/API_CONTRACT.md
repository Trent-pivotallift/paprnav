# paprnav API Contract

Last updated: 2026-06-16

This is the interim human-readable API contract until FastAPI Pydantic schemas and OpenAPI-generated frontend types are implemented. Per D011, the backend OpenAPI document becomes the source of truth once endpoint schemas exist.

## Contract Rules

- API routes are versioned under `/api/v1`.
- Request and response bodies are JSON unless a route explicitly accepts multipart upload.
- IDs are opaque strings generated by the backend.
- Timestamps use ISO 8601 strings.
- N-numbers are accepted as user input but returned with both raw and normalized forms where relevant.
- Protected routes require the MVP session cookie from D010.
- Error responses should use a stable shape:

```json
{
  "error": {
    "code": "string",
    "message": "string",
    "details": {}
  }
}
```

## Auth Assumptions

MVP auth is FastAPI-owned session auth with secure HTTP-only cookies. The frontend should call auth endpoints with credentials included.

### Register

`POST /api/v1/auth/register`

Request:

```json
{
  "email": "owner@example.com",
  "name": "Sample Owner",
  "password": "secret-password"
}
```

Response:

```json
{
  "user": {
    "id": "usr_123",
    "email": "owner@example.com",
    "name": "Sample Owner"
  }
}
```

### Login

`POST /api/v1/auth/login`

Request:

```json
{
  "email": "owner@example.com",
  "password": "secret-password"
}
```

Response:

```json
{
  "user": {
    "id": "usr_123",
    "email": "owner@example.com",
    "name": "Sample Owner"
  }
}
```

## Aircraft

### List Aircraft

`GET /api/v1/aircraft`

Returns aircraft visible to the authenticated user. Owner users see owned aircraft; maintenance users see assigned client aircraft.

Response:

```json
{
  "aircraft": [
    {
      "id": "ac_123",
      "nNumber": "N123AB",
      "nNumberNormalized": "N123AB",
      "make": "Cessna",
      "model": "172R",
      "serialNumber": "17280001",
      "lastLogEntryDate": "2026-01-15",
      "complianceStatus": "needs_review"
    }
  ]
}
```

### Create Aircraft

`POST /api/v1/aircraft`

Request:

```json
{
  "nNumber": "N123AB",
  "make": "Cessna",
  "model": "172R",
  "serialNumber": "17280001",
  "engineMake": "Lycoming",
  "engineModel": "IO-360",
  "propellerMake": "McCauley",
  "propellerModel": "1A170"
}
```

Response: the created aircraft object.

### Get Aircraft

`GET /api/v1/aircraft/{aircraftId}`

Response: one aircraft object with component identity fields.

### Update Aircraft

`PATCH /api/v1/aircraft/{aircraftId}`

Request: partial aircraft fields.

Response: updated aircraft object.

## Uploads And Ingestion

Per D013, files use a storage abstraction. Local development stores files on disk; production uses S3 or compatible object storage. API responses should expose file/job IDs and retrieval URLs, not local filesystem paths or raw bucket internals.

### Upload Logbook File

`POST /api/v1/aircraft/{aircraftId}/uploads`

Content type: `multipart/form-data`

Fields:

- `file`: PDF, JPG, JPEG, or PNG
- `section`: optional initial section hint

Response:

```json
{
  "upload": {
    "id": "upl_123",
    "aircraftId": "ac_123",
    "originalFilename": "logbook.pdf",
    "contentType": "application/pdf",
    "fileSizeBytes": 1234567,
    "sha256": "hex-hash",
    "status": "stored",
    "downloadUrl": "/api/v1/uploads/upl_123/download"
  }
}
```

### Download Uploaded File

`GET /api/v1/uploads/{uploadId}/download`

Returns the stored original file when the authenticated user can access the linked aircraft.

### Create Ingestion Job

`POST /api/v1/uploads/{uploadId}/ingestion-jobs`

Response:

```json
{
  "job": {
    "id": "job_123",
    "uploadId": "upl_123",
    "aircraftId": "ac_123",
    "status": "queued",
    "pageExtractionStatus": "queued",
    "ocrStatus": "not_started",
    "verificationStatus": "not_started"
  }
}
```

### Get Ingestion Job

`GET /api/v1/ingestion-jobs/{jobId}`

Response: job status plus page summaries when available.

### Confirm Page Order And Completeness

`POST /api/v1/ingestion-jobs/{jobId}/page-verification`

Request:

```json
{
  "pageOrder": ["page_1", "page_2", "page_3"],
  "isComplete": true,
  "notes": ""
}
```

Response: updated verification state.

### Submit OCR Correction

`POST /api/v1/ingestion-jobs/{jobId}/ocr-corrections`

Request:

```json
{
  "pageId": "page_1",
  "ocrTextSpanId": "span_123",
  "correctedText": "Corrected maintenance text",
  "correctionReason": "low_confidence",
  "notes": ""
}
```

Response: created correction.

## AD Ingestion And Review Boundary

The first implementation can keep AD ingestion worker commands internal. When exposed through the API, endpoints should follow the same `/api/v1` contract and return reviewed/approved AD records, extraction review tasks, and matching worklist items. AD source persistence, extraction metadata, review thresholds, and future AWS mapping are defined in `.ai/AD_INGESTION_MVP_SPEC.md` and D015.
